## Part B - Execution Instructions and Key Results (AA subset only)

### Objective

- (1) Words (not numbers) with total count exactly `1000`.
- (2) Word pairs (not numbers) with total count exactly `1000`, with neighbor distance `m <= 2`.
- (3) Top-100 most frequent words (not numbers).
- (4) Top-100 most frequent pairs where first token is a number and second is a word, with `m <= 2`.

### Dataset scope

- Input path: `./Wikipedia-En-41784-Articles/AA/*`
- Number of input files: `100`

### Solution strategy

- Implemented in one MapReduce job/class: `HadoopPartB`.
- No post-processing outside map/reduce for filtering/top-100 logic.
- One reducer (`setNumReduceTasks(1)`) is used so global top-100 is exact.

### Pair definition clarification

- Pair generation is ordered and forward-only inside each line after token filtering.
- For token index `i`, neighbors are `i+1` and `i+2` (if present), i.e., distance `m in {1,2}`.
- No backward pairs and no cross-line pairs are generated.
- For Task (4), only pairs matching `number:word` in that order are counted.

### Build commands (from repository root)

Compile Java sources:

```bash
javac -cp "$(hadoop classpath)" problem_1/*.java
```

Create/recreate jar:

```bash
jar -cvf problem_1/HadoopWordCount.jar -C problem_1 .
```

### Run commands (AA only)

```bash
rm -rf problem_1/HadoopPartB_Output_AA
/usr/bin/time -p hadoop jar problem_1/HadoopWordCount.jar HadoopPartB "./Wikipedia-En-41784-Articles/AA/*" problem_1/HadoopPartB_Output_AA
```

### Output format (`part-r-00000`)

- `TASK1\t<word>\t<count>` for (1)
- `TASK2\t<word1:word2>\t<count>` for (2)
- `TASK3\t<word>\t<count>` for (3) top-100 words
- `TASK4\t<number:word>\t<count>` for (4) top-100 number-word pairs

**Result counts**

**Output examples**

### Dataset scope and method

- Job benchmarked: `HadoopPartB` (the combined Part B solution).
- Input grows cumulatively by one subdirectory per step: `AA`, `AA+AB`, ..., `AA+...+AK`.
- For each step, one full MapReduce execution is timed with `/usr/bin/time -p`.
- Output folder per step: `problem_1/HadoopPartB_Output_<LABEL>`.
- Generated output contents are ignored via `problem_1/.gitignore`, while each output folder keeps a tracked empty `.gitignore` placeholder.

### Notes and scalability conclusions

- Runtime growth is close to linear with input size in this local setup.
- From `100` to `1060` files, runtime increases from `18.83s` to `163.29s` (~`8.67x`), while input size increases ~`10.6x`.
- This indicates good practical scalability for the implemented combined single-job approach.
- Small deviations from perfect linearity are expected due to LocalJobRunner overheads, filesystem caching, and changing token distributions.
- Global top-100 correctness is preserved at all scales because a single reducer is used.
